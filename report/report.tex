\section{Introduction}

\subsection{Project Overview}
\textbf{TennisLytics} is a full-stack web-based Tennis Analytics Dashboard built using the \textbf{MERN stack} (MongoDB, Express.js, React.js, Node.js). The application provides a comprehensive platform for exploring, analyzing, and visualizing professional tennis match data sourced from the ATP (Association of Tennis Professionals) tour.

The system ingests real-world ATP match data from CSV files, stores it in a normalized MongoDB database with two primary collections (\texttt{Players} and \texttt{Matches}), and exposes a rich RESTful API powered by Express.js. The React-based frontend delivers an interactive, responsive single-page application (SPA) with data visualizations using \textbf{Recharts}, a modern charting library.

\subsection{Motivation}
Tennis generates vast amounts of structured data --- player rankings, match scores, surface preferences, head-to-head records, and set-by-set breakdowns. Existing platforms present this data in static tables with limited interactivity. TennisLytics addresses this gap by offering:

\begin{itemize}[noitemsep]
    \item Interactive player search with \textbf{real-time filtering}
    \item Surface-wise win-percentage analytics using \textbf{MongoDB aggregation pipelines}
    \item Head-to-head comparison with \textbf{searchable dropdowns}
    \item Match exploration by type (tiebreakers, sweeps, close sets, dominant wins)
    \item Rivalry detection using \textbf{advanced aggregation} ($\$$addFields, $\$$cond, $\$$group)
    \item Tournament and player demographic analytics
\end{itemize}

\subsection{Scope}
The project demonstrates proficiency in:
\begin{enumerate}[noitemsep]
    \item \textbf{Database Design}: Normalized schemas with embedded documents and ObjectId references
    \item \textbf{MongoDB Query Mastery}: 20+ distinct query operators and aggregation stages
    \item \textbf{REST API Architecture}: 31 API endpoints across two route modules
    \item \textbf{Frontend Engineering}: Component-based React architecture with client-side routing
    \item \textbf{Data Visualization}: Interactive charts (PieChart, BarChart) with Recharts
    \item \textbf{Deployment Readiness}: Architecture designed for Vercel (frontend) + MongoDB Atlas (database)
\end{enumerate}

\section{Problem Statement}

\subsection{Background}
The ATP tour generates over 3,000 professional matches annually across multiple surfaces (Hard, Clay, Grass), tournament tiers (Grand Slams, Masters, Challengers), and rounds. This data, while publicly available in raw CSV formats, is difficult to explore without specialized tools.

\subsection{Problem Definition}
Design and implement a \textbf{full-stack web application} that:

\begin{enumerate}
    \item \textbf{Ingests} raw ATP match data from CSV files into a MongoDB database with properly normalized schemas (Player and Match collections with ObjectId references).

    \item \textbf{Exposes} a comprehensive RESTful API that demonstrates mastery of MongoDB query operations, including:
    \begin{itemize}[noitemsep]
        \item Basic CRUD operations (\texttt{find}, \texttt{create}, \texttt{update}, \texttt{delete})
        \item Query operators (\texttt{\$regex}, \texttt{\$or}, \texttt{\$ne}, \texttt{\$gte}, \texttt{\$lte}, \texttt{\$in})
        \item Array query operators (\texttt{\$elemMatch}, \texttt{\$exists}, \texttt{\$not}, \texttt{\$all})
        \item Aggregation pipeline stages (\texttt{\$match}, \texttt{\$group}, \texttt{\$sort}, \texttt{\$limit}, \texttt{\$lookup}, \texttt{\$unwind}, \texttt{\$project}, \texttt{\$addFields}, \texttt{\$facet}, \texttt{\$cond}, \texttt{\$size}, \texttt{\$filter}, \texttt{\$concat}, \texttt{\$toString}, \texttt{\$slice}, \texttt{\$addToSet}, \texttt{\$push})
        \item Utility methods (\texttt{countDocuments}, \texttt{distinct}, \texttt{populate}, \texttt{lean})
    \end{itemize}

    \item \textbf{Renders} an interactive, responsive frontend with:
    \begin{itemize}[noitemsep]
        \item Sortable and filterable player dashboard
        \item Player detail pages with surface-wise analytics and PieCharts
        \item Searchable head-to-head comparison tool
        \item Tournament analytics with BarCharts
        \item Match explorer with 6 query-based filters
        \item Player demographic insights (hand distribution, country breakdown)
        \item Top rivalries leaderboard
    \end{itemize}
\end{enumerate}

\newpage
\subsection{Objectives}
\begin{enumerate}[noitemsep]
    \item Parse and seed 11,190 match records involving 1,477 players from ATP CSV data
    \item Implement 31 REST API endpoints showcasing diverse MongoDB query patterns
    \item Build 8 distinct frontend views with data visualizations
    \item Ensure deployment readiness for Vercel (frontend) and MongoDB Atlas (backend)
\end{enumerate}

\subsection{MongoDB Schema Design}

\subsubsection{Player Collection}
\begin{lstlisting}[language=JavaScript, caption={Player Schema (Mongoose)}]
const playerSchema = new mongoose.Schema({
    name:        { type: String, required: true },
    nationality: { type: String, required: true },
    rank:        { type: Number },
    hand:        { type: String, enum: ['R', 'L', 'U'], default: 'R' },
}, { timestamps: true });
\end{lstlisting}

\subsubsection{Match Collection (with Embedded Sets)}
\begin{lstlisting}[language=JavaScript, caption={Match Schema with Embedded Set Sub-document}]
const setSchema = new mongoose.Schema({
    setNum:      { type: Number, required: true },
    winnerScore: { type: Number, required: true },
    loserScore:  { type: Number, required: true },
    tiebreak:    { type: Boolean, default: false },
}, { _id: false });

const matchSchema = new mongoose.Schema({
    tourney_name: { type: String, required: true },
    surface:      { type: String, required: true },
    winner_id:    { type: ObjectId, ref: 'Player', required: true },
    loser_id:     { type: ObjectId, ref: 'Player', required: true },
    sets:         [setSchema],      // Embedded array of sub-documents
}, { timestamps: true });
\end{lstlisting}

\subsubsection{Database Indexing Strategy}
To optimize query performance across the 11,000+ match records and 1,400+ players, several indexes were implemented in the schema definitions. Simple single-field indexes were added on frequently filtered and sorted fields such as \texttt{rank}, \texttt{nationality}, \texttt{winner\_id}, and \texttt{loser\_id}. To support efficient text search, a text index was applied to the player \texttt{name} field. Furthermore, compound indexes (e.g., \texttt{\{ winner\_id: 1, loser\_id: 1 \}}) were created to optimize complex aggregations like head-to-head comparisons, and multikey indexes on embedded document fields (e.g., \texttt{sets.tiebreak} and \texttt{sets.loserScore}) were utilized to accelerate array-based \texttt{\$elemMatch} queries.

\section{Conclusion}

\subsection{Summary}
The \textbf{TennisLytics} project successfully demonstrates a production-grade full-stack web application built on the MERN stack. The key accomplishments include:

\begin{enumerate}
    \item \textbf{Database Design}: Two Mongoose schemas (Player and Match) with embedded sub-documents (sets array with tiebreak tracking) and ObjectId cross-references, showcasing both embedding and referencing patterns in MongoDB.

    \item \textbf{Comprehensive MongoDB Usage}: 31 API endpoints exercising \textbf{25+ distinct MongoDB operators}, spanning basic CRUD, query operators (\texttt{\$regex}, \texttt{\$or}, \texttt{\$ne}, \texttt{\$gte}, \texttt{\$lte}), array operators (\texttt{\$elemMatch}, \texttt{\$not}, \texttt{\$exists}), expression operators (\texttt{\$expr}, \texttt{\$size}), and advanced aggregation stages (\texttt{\$facet}, \texttt{\$lookup}, \texttt{\$addFields}, \texttt{\$cond}, \texttt{\$filter}, \texttt{\$concat}).

    \item \textbf{Data Pipeline}: A robust seed script that parses raw ATP CSV data (11,190 rows), normalizes tennis score strings (handling tiebreaks, retirements, walkovers), and performs batch upserts into MongoDB.

    \item \textbf{Frontend}: 8 interactive views with data visualizations (PieCharts, BarCharts), searchable components, sortable tables, and a premium dark-mode design system with glassmorphism and micro-animations.

    \item \textbf{Feature Depth}: Player management (CRUD), surface analytics, head-to-head comparison, tournament leaderboards, match explorer (6 filter types), rivalry detection, and player demographic insights.
\end{enumerate}

    \item \textbf{Advanced Analytics}: Add Elo rating calculations, win prediction models using match statistics (aces, break points, serve percentages).

    \item \textbf{Testing}: Add unit tests (Jest) for API endpoints and integration tests (Cypress) for frontend workflows.

\end{enumerate}
